# 整体架构地图
这一章解决的问题：先建立 Valhalla 的“请求链路 + 数据链路”地图。
读完这一章，你应该能回答：route 请求经过哪些模块、离线 tiles 怎么来、每个模块第一次应该读哪里。
建议先读：`src/tyr/actor.cc`、`src/loki/route_action.cc`、`src/thor/route_action.cc`、`src/odin/worker.cc`。
不建议一开始读：具体 A* 模板细节、所有 costing 参数、所有 tile 构建 stage。

你可以先把 Valhalla 理解成两条链路：一条是离线把 OSM 等数据构造成 routing tiles；另一条是在线收到请求后，用这些 tiles 完成位置关联、路径搜索和结果生成。`actor_t::route` 直接给了最短主线：`ParseApi` 解析请求，然后调用 Loki、Thor、Odin。证据是 `src/tyr/actor.cc:97-118`。

## 在线请求主链路
```text
Client
  -> Tyr: 解析请求、编排 worker
  -> Loki: 校验 locations/costing，并把坐标关联到图
  -> Thor: 搜索路径、matrix、isochrone 或 trace route
  -> Sif: Thor 搜索时使用 DynamicCost 计算边和转向代价
  -> Odin: 把路径转成 maneuver/narrative
  -> Tyr: 序列化 JSON/PBF
```

这不是凭模块名猜的：`actor_t::route` 的注释和调用顺序写明“check request and locate”、“route between the locations”、“get directions back and serialize”。证据是 `src/tyr/actor.cc:110-118`。

## 离线构建链路
`valhalla_build_tiles` 调用 `build_tile_set`；`build_tile_set` 依次处理 ParseWays、ParseRelations、ParseNodes、ConstructEdges、Build、Enhance 等阶段。第一次读时只要知道：Mjolnir 产出 Baldr 能读的 GraphTile。证据是 `src/mjolnir/valhalla_build_tiles.cc:98` 和 `src/mjolnir/util.cc:631-790`。

## 本章小结
先抓住两条链路：在线 Tyr->Loki->Thor->Odin，离线 Mjolnir->Baldr tiles。Sif 不是单独处理请求的 worker，而是 Thor 搜索时使用的代价模型。

## 下一步读什么
如果只关心 route，读 `03-一次route请求如何跑完.md`；如果要理解图数据，读 Baldr 和 Mjolnir；如果要改 ETA 或偏好，读 Sif 和 Thor。
